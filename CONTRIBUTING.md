## 행동 강령

커뮤니티의 모든 상호작용에서 서로 존중하고 건설적으로 소통해주세요.

## 시작하기

1. 저장소를 포크합니다
2. 포크한 저장소를 클론합니다:
   ```bash
   git clone https://github.com/YOUR_USERNAME/ssag.git
   cd ssag
   ```
3. 의존성을 설치합니다:
   ```bash
   npm install
   ```
4. 새로운 브랜치를 생성합니다:
   ```bash
   git checkout -b feature/기능명
   ```

## 개발 프로세스

1. 피처 브랜치에서 변경사항을 작업합니다
2. 로컬에서 변경사항을 테스트합니다:
   ```bash
   npm run dev
   npm run lint
   npm run build
   ```
3. [커밋 컨벤션](#커밋-컨벤션)을 따라 커밋합니다
4. 포크한 저장소에 푸시합니다
5. Pull Request를 생성합니다

## 커밋 컨벤션

[Conventional Commits](https://www.conventionalcommits.org/) 규칙을 따릅니다.

### 형식

```
<타입>(<범위>): <제목>

<본문>

<꼬리말>
```

### 타입

- **feat**: 새로운 기능 추가
- **fix**: 버그 수정
- **docs**: 문서 변경
- **style**: 코드 의미에 영향을 주지 않는 변경사항 (공백, 포맷팅 등)
- **refactor**: 버그 수정이나 기능 추가가 아닌 코드 변경
- **perf**: 성능 개선을 위한 코드 변경
- **test**: 테스트 추가 또는 수정
- **chore**: 빌드 프로세스나 보조 도구 변경

### 예시

```bash
feat(router): about 페이지 추가
fix(home): 모바일 환경에서 네비게이션 이슈 해결
docs(readme): 설치 방법 업데이트
style(app): prettier로 코드 포맷팅
refactor(api): 데이터 페칭 로직 단순화
perf(home): 이미지 로딩 최적화
test(app): 라우팅 단위 테스트 추가
chore(deps): react 버전 19로 업데이트
```

### 범위 (선택사항)

영향을 받는 컴포넌트나 모듈의 이름을 작성합니다 (예: `router`, `home`, `api` 등).

### 제목

- 명령형 현재 시제 사용: "추가한다" (X) → "추가" (O)
- 첫 글자를 대문자로 작성하지 않습니다
- 마침표(.)를 끝에 붙이지 않습니다
- 50자 이내로 작성합니다

### 본문 (선택사항)

- 무엇을, 왜 변경했는지 설명 (어떻게 보다는)
- 72자마다 줄바꿈
- 제목과 본문 사이 한 줄 비우기

### 꼬리말 (선택사항)

- 이슈 참조: `Closes #123` 또는 `Fixes #456`
- 주요 변경사항: `BREAKING CHANGE: 설명`

## Pull Request 프로세스

1. 코드가 프로젝트의 코딩 규칙을 따르는지 확인합니다
2. 필요한 경우 문서를 업데이트합니다
3. 해당되는 경우 테스트를 추가합니다
4. 모든 테스트가 통과하고 린트 에러가 없는지 확인합니다
5. Pull Request 템플릿을 완전히 작성합니다
6. 관련 이슈를 연결합니다
7. 코드 리뷰를 기다리고 피드백을 반영합니다

## 코딩 규칙

### JavaScript/React

- Hook을 사용한 함수형 컴포넌트 사용
- ESLint 설정을 따릅니다
- 의미있는 변수명과 함수명 사용
- 자체 설명이 가능한 코드 작성
- 복잡한 로직에 주석 추가

### 파일 구조

- 컴포넌트: `PascalCase.jsx`
- 유틸리티: `camelCase.js`
- 파일을 집중적이고 단일 목적으로 유지

### 모범 사례

- 컴포넌트를 작고 집중적으로 유지
- 재사용 가능한 로직은 커스텀 훅으로 추출
- 네비게이션에 React Router 사용
- 에러를 적절히 처리
- 깨끗하고 읽기 쉬운 코드 작성
